## DoR e DoD

### Definition of Ready (DoR)

A Definition of Ready (DoR) é um conjunto de critérios que um item ou uma User Story do backlog deve atender para ser considerado pronto para iniciar o desenvolvimento. Assim que um item atende a esses critérios, ele pode ser puxado para a Sprint.

Para que um item seja considerado **Ready**, ele deve atender aos seguintes critérios:

- A história de usuário está no formato:  
  *"Eu como [ator], devo ser capaz de [ação], para que [objetivo]"*?

- O requisito não possui ambiguidade?  
  Ou seja, não há múltiplas interpretações para desenvolvedores e stakeholders.

- Os stakeholders definiram os critérios de aceitação?  
  Devem estar claros os comportamentos e ações esperados no software.

- Os stakeholders forneceram o protótipo da interface?  
  A empresa possui apoio de designer e assessoria de imagem, sendo necessário que novas telas sejam aprovadas previamente.

- O requisito foi validado pelo coordenador de requisitos?  
  Deve estar revisado e em conformidade com padrões de qualidade, incluindo clareza, viabilidade técnica e ausência de ambiguidades.

### Definition of Done (DoD)

A Definition of Done (DoD) define os critérios necessários para que um item seja considerado concluído.

Para que um item seja considerado **Done**, ele deve atender aos seguintes critérios:

- A interface está de acordo com o protótipo aprovado pelos stakeholders?  
  Deve seguir o design definido pela equipe de design.

- Os critérios de aceitação foram atendidos?  
  Todas as funcionalidades devem cumprir os comportamentos esperados definidos previamente.

- O backend foi integrado ao frontend (quando necessário)?  
  A integração deve estar funcionando corretamente.

- O código passou por revisão de pares?  
  Pelo menos dois revisores devem aprovar o código, garantindo qualidade e aderência aos padrões do projeto.

- O backend foi validado pelo coordenador Daniel Rodrigues (quando aplicável)?  
  Deve garantir segurança e integridade na manipulação dos dados.

- O código foi validado pelo analista de QA João Pedro?  
  Deve possuir cobertura de testes de, no mínimo, 80%.

- A funcionalidade foi integrada ao branch principal sem conflitos?  
  Não deve interferir em outras funcionalidades ou no funcionamento geral do sistema.

- A funcionalidade está disponível em ambiente de homologação ou produção?  
  Deve estar acessível para validação pelos stakeholders após a entrega.
